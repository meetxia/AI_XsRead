# 会员体系（VIP + 激活码）— 需求文档

> 文档日期：2026-05-17  
> 项目：MOMO小说（AI-XsRead）  
> 状态：草案（基于 2026-05-17 lead 与产品方对齐结果）

## 1. 业务背景

平台目前所有书籍对所有用户开放。需要新增"VIP 内容"维度，让一部分书只有付费会员可读。
付费方式不接入直接支付，而是引导到第三方发卡平台购买激活码，用户拿到激活码后在站内激活。

## 2. 功能边界

### 范围内
- 书籍 VIP 标记（整本维度）
- 用户会员等级与到期日字段
- 激活码生成 / 激活 / 作废 / 导出
- 三个激活入口：个人中心、注册页、阅读墙
- 游客与免费用户对 VIP 书的"试读 1500 字 + 会员墙"体验
- 管理员后台：批次管理、激活码生成（单个/批量，单次最多 1000）、CSV 导出、激活记录查询、人工停用会员
- 审计日志：生成日志、激活日志
- 客服联系入口（QQ / 微信号 / 二维码）

### 范围外（不在本期）
- 章节级 VIP（先做整本）
- 试读"前 N 章" 模式
- 直接在线支付
- 邀请好友送会员
- 多产品线（漫画 / 听书）
- 退款流程（仅人工后台作废 + 加时长方式处理）

## 3. 角色与术语

| 角色 | 说明 |
|---|---|
| 游客 | 未登录访问者，可看免费书全文、可看 VIP 书前 1500 字试读 |
| 普通用户 | 已登录但未激活会员 / 会员已过期 |
| 月卡会员 / 年卡会员 / 永久会员 | 已激活对应等级且未过期 |
| Admin | 管理后台用户，所有 admin 都可生成激活码 |
| 激活码（activation code）| 16 位大写字母+数字，一码一用 |
| 批次（batch）| 一次"生成 N 个码"产生的逻辑分组，可整体导出 |

## 4. 用户故事

### US-1 游客浏览免费书 ✅
作为游客，我打开任意免费小说详情页，点"开始阅读"，能看到全部内容。

### US-2 游客遇到 VIP 书
作为游客，我打开 VIP 书详情页，看到"会员可读"角标和"开通会员后阅读"按钮。
点开始阅读，能看到前 1500 字试读内容，下方出现会员墙：列出三档权益、立即激活按钮、"已有激活码"链接。

### US-3 普通用户激活会员（个人中心入口）
作为登录用户，我进入个人中心 → 会员中心，输入 16 位激活码，提交后：
- 月卡：到期日 = max(now, 当前到期日) + 30 天
- 年卡：到期日 = max(now, 当前到期日) + 365 天
- 永久：vip_level = 3，到期日 = 2099-12-31
立即可读全部 VIP 书。

### US-4 注册时激活
作为新用户，我在注册表单底部展开"我有激活码（可选）"，填入激活码后提交注册。
注册成功后立即是会员状态。注册和激活原子完成（激活码失败不影响注册成功，但提示用户后续可在个人中心激活）。

### US-5 试读墙激活
作为游客，我读完 VIP 书的 1500 字试读，点"立即激活"会跳到登录/注册页（携带 returnUrl 回到原书）；
登录后再点"已有激活码"，输入激活后立即解锁全文继续阅读。

### US-6 会员到期前提醒
作为会员，我的到期日小于等于 7 天时，个人中心显示"还剩 X 天"红色提醒。
首页/详情页**不弹窗骚扰**。永久会员永远不显示倒计时。

### US-7 续期叠加
作为已是月卡会员，剩余 10 天，再激活一张月卡：到期日变为剩余 40 天。
月卡 → 再激活年卡：到期日 += 365 天，但 vip_level 升为 2。
任何卡 → 激活永久卡：vip_level = 3，到期日设为 2099-12-31。

### US-8 admin 生成激活码
作为 admin，我进入"激活码管理 → 新建批次"页：
- 选择类型（月卡 / 年卡 / 永久）
- 输入数量（1-1000，超过要二次确认）
- 输入备注（如"2026-05 双 11 营销"）
- 点"生成" → 系统创建批次 + N 个激活码
- 立即跳转批次详情页，提供"复制全部"和"导出 CSV"按钮

### US-9 admin 导出
作为 admin，我在批次详情页或批次列表页都能"导出 CSV"，得到原始可见的激活码列表。

### US-10 admin 作废未激活码
作为 admin，我能在批次详情里勾选"未使用"的码 → 批量作废。
作废后这批码立即不可激活，但会保留在数据库供审计。

### US-11 admin 停用会员
作为 admin，在用户管理页能看到用户的会员状态，可点"停用会员"立即把该用户置为非会员（vip_expires_at 置为过去时间，记录操作日志）。

### US-12 admin 查激活记录
作为 admin，能按用户 / 时间 / 批次过滤查激活记录。

### US-13 客服联系
所有用户在个人中心、会员墙、注册页都能看到"客服联系"卡片，展示 QQ 号、微信号、微信二维码图片。

## 5. 业务规则

### R-1 一码一用
任何激活码激活成功后立刻置 status=used，不再可用。

### R-2 续期延长
| 当前状态 | 激活月卡 | 激活年卡 | 激活永久 |
|---|---|---|---|
| 非会员或已过期 | 30 天，level=1 | 365 天，level=2 | 永久，level=3 |
| 月卡未过期 | 剩余 + 30 天 | 剩余 + 365 天，level=2 | 永久，level=3 |
| 年卡未过期 | 剩余 + 30 天 | 剩余 + 365 天 | 永久，level=3 |
| 永久 | 拒绝（已是永久） | 拒绝 | 拒绝（已是永久） |

注：vip_level 永远以"已激活的最高等级"为准；时长一律按所激活卡的天数叠加。

### R-3 限流
单用户对 `/api/user/membership/activate` 接口：
- 1 小时内最多尝试 5 次（成功也计数）
- 超出返回 429，提示"激活过于频繁，请稍后重试"

### R-4 admin 单次生成
- 1-100：直接生成
- 101-1000：前端必须二次确认弹窗（"确定一次性生成 X 个？"）
- 0 或 >1000：拒绝

### R-5 激活码字符集
- 长度：16 位
- 字符集：`A-Z 0-9` 共 36 个，剔除 `O 0 I 1 B 8` 共 30 个备选
- 显示格式：4-4-4-4 分组（如 `K7H2-P3MX-9YRT-4QFW`）
- 输入接收：去掉所有空格和 `-`，强制大写后比对

### R-6 安全
- 激活码入库前用项目 JWT_SECRET 做 AES-256-GCM 加密
- 同时存 SHA-256(明文) 作 lookup_hash，激活时按 hash 查询
- 后台展示需要权限校验后才解密返回明文

### R-7 试读字数
- 游客 / 非会员打开 VIP 书章节内容接口，仅返回前 1500 字 + `truncated: true`
- 普通用户已激活则返回完整内容
- 免费书（is_vip=0）任何人都返回完整内容

### R-8 客服信息维护
管理后台单独有"系统配置 → 客服联系"页面，可编辑 QQ 号、微信号，上传二维码图片。前端通过公开接口 `/api/system/contact` 拉取。

## 6. 数据模型变更

### 新增字段
- `users.vip_level TINYINT(1) DEFAULT 0`（0 普通 / 1 月卡 / 2 年卡 / 3 永久）
- `users.vip_expires_at DATETIME NULL`
- `users.vip_status TINYINT(1) DEFAULT 1`（1 正常 / 0 被管理员停用）
- `novels.is_vip TINYINT(1) DEFAULT 0`

### 新增表
- `code_batches`：批次表
- `activation_codes`：激活码表
- `code_redemption_logs`：激活记录（审计）
- `code_admin_logs`：管理员操作日志（生成 / 作废 / 停用会员）
- `system_contact`：客服信息（单行表）

字段细节见 `design.md`。

## 7. API 概览

### 用户端
- `GET /api/user/membership` 查当前会员状态
- `POST /api/user/membership/activate { code }` 激活
- `GET /api/system/contact` 公开客服信息
- 既有 `POST /api/auth/register` 扩展 `activation_code` 可选字段

### 管理后台
- `POST /api/admin/codes/batches` 创建批次（生成 N 个码）
- `GET /api/admin/codes/batches` 批次列表
- `GET /api/admin/codes/batches/:id` 批次详情（含码列表）
- `GET /api/admin/codes/batches/:id/export` CSV 导出
- `POST /api/admin/codes/void` 批量作废未使用码
- `GET /api/admin/codes/redemptions` 激活记录查询
- `POST /api/admin/users/:id/membership/disable` 停用某用户会员
- `GET /api/admin/system/contact` / `PUT /api/admin/system/contact` 维护客服信息

### VIP gating
既有 `GET /api/novels/:id`、`GET /api/chapters/:id`、`GET /api/novels/:id/pages` 增加 `is_vip` 字段返回 + 试读截断逻辑。

## 8. 验收标准

### 用户侧
- [ ] 游客打开免费书 → 全文可读
- [ ] 游客打开 VIP 书 → 试读 1500 字 + 会员墙
- [ ] 用户激活月卡 → vip_expires_at 正确（now + 30 天）
- [ ] 月卡续期 → 剩余天数正确叠加
- [ ] 永久会员激活其他码 → 拒绝
- [ ] 注册时激活码失败 → 用户仍创建成功，提示文案明确
- [ ] 1 小时第 6 次激活 → 429
- [ ] 会员到期前 7 天 → 个人中心显示倒计时

### 管理后台侧
- [ ] 单次生成 50 个 → 即时返回 + 可导出 CSV
- [ ] 单次生成 800 个 → 二次确认 → 异步反馈进度（或同步等待 + loading）
- [ ] 单次生成 1500 个 → 接口拒绝
- [ ] 导出 CSV 任意时刻可重新下载（不仅生成时）
- [ ] 作废未使用码 → 该批量码立即不可激活
- [ ] 作废已使用码 → 拒绝
- [ ] 停用某用户会员 → 该用户立刻无法读 VIP 书
- [ ] 所有操作记录在 code_admin_logs

### 安全
- [ ] DB 直接 SELECT activation_codes.code_encrypted 看不出明文
- [ ] 普通用户 token 调 admin 接口 → 403
- [ ] 激活码激活失败次数过多 → 429

## 9. 风险与注意事项

1. **第三方发卡平台跑路**：用户已付钱但拿不到激活码 → 客服处理，与平台无关
2. **激活码泄露**：如 admin 账号被攻破 → 立即作废全部未使用码 + 改 admin 密码
3. **存量数据**：现有 34 本书全部 is_vip=0；现有所有用户 vip_level=0、vip_expires_at=NULL
4. **试读截断口径**：1500 字按"中文字符 + 英文单词"如何统一计数 → 设计稿统一为"按 String.length 截断 1500 个字符"，简单直观
5. **永久会员展示**：永久会员到期日 2099-12-31 不要直接展示给用户，应在前端识别 vip_level=3 显示为"永久会员"
6. **续期边界**：测试用户已过期 1 天，再激活月卡，应该是 now + 30 天还是 (过期日) + 30 天？设计稿统一为：**`max(now, vip_expires_at) + days`**

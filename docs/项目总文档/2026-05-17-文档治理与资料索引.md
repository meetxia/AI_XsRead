# 文字之境（AI-XsRead）文档治理与资料索引

文档日期：2026-05-17

## 1. 目的

本项目的文档已经发生明显迁移和过期问题。本文档用于说明：

1. 哪些目录是当前主口径
2. 哪些资料已过期但仍可参考
3. 新文档应该放在哪里
4. 以后如何避免再次口径漂移

## 2. 当前主文档目录

### 正式主文档
- `docs/01-核心文档/`
- `docs/02-快速启动指南/`
- `docs/项目总文档/`
- `docs/AI完成后报告文档/`

### 参考与遗留资料
- `docx/`
- `docx/backup_before_organize/`
- `docs/历史研究归档/`

说明：
- `docs/` 目录已是新的正式文档承载位置。
- `docx/` 更适合作为历史备份和迁移参考，不应继续作为当前正式口径的主要入口。
- `docs/历史研究归档/` 用于保留旧阶段研究材料，不能作为当前开发、测试、排期、上线判断依据。

## 3. 当前应优先信任的资料类型

按可信度由高到低建议这样理解：

1. 当前代码实现
2. `docs/AI完成后报告文档/` 下最新日期的交付报告
3. `docs/项目总文档/` 中的当前总览、索引、清单类文档
4. `docs/01-核心文档/` 与 `docs/02-快速启动指南/` 中已更新的文档
5. `docx/backup_before_organize/` 中的历史文件
6. `docs/历史研究归档/` 中的旧阶段研究材料
7. `docx/` 下未迁移或未更新的旧文件

## 4. 明显过期的文档口径

以下类型内容应视为明显过期或至少需要重新核对：

1. 旧版快速启动说明中出现的老端口、老路径、老数据库名
2. 模板化 README 中沿用的通用 Vue/Vite 初始文案
3. 旧版 API 设计文档中未包含新增阅读、段评、划线、关注作者、勋章等能力的内容
4. 旧版数据库文档中缺少迁移脚本和新增表域的内容
5. 旧版“项目技术分析报告”中对当前模块边界的描述
6. 2025-10-27 项目经理研究报告中的“85% 完成度”“V1.1 / V1.5 / V2.0”等旧阶段判断

## 5. 建议本轮新建文档清单

本轮建议以 `2026-05-17` 为统一日期前缀，形成以下核心索引文档：

1. `docs/项目总文档/2026-05-17-项目现状总览.md`
2. `docs/项目总文档/2026-05-17-启动与环境配置总览.md`
3. `docs/项目总文档/2026-05-17-接口与路由速查总表.md`
4. `docs/项目总文档/2026-05-17-数据库与迁移现状说明.md`
5. `docs/项目总文档/2026-05-17-文档治理与资料索引.md`

## 6. 建议保留但降级为历史参考的内容

以下资料可保留，但建议在正式文档中降级说明：

- 旧版启动手册
- 旧版数据库 SQL 汇总
- 旧版项目技术分析报告
- 旧版 bug 检查与优化报告
- 旧版阶段性功能交付报告
- 2025-10-27 项目经理研究报告

这些内容并非完全无用，但它们更适合用于“演进回溯”，不适合作为当前实现说明的主入口。

其中，旧项目经理研究报告已经迁移到：

- `docs/历史研究归档/项目经理/`

原目录 `docs/本项目研究报告-项目经理/` 仅保留入口说明，提醒读者不要继续把旧报告作为当前口径。

## 7. 文档冲突处理原则

当文档口径与代码实现冲突时，采用以下顺序：

1. 当前代码实现
2. 最新日期的交付报告
3. 当前总览与索引文档
4. 旧正式文档
5. 迁移前备份

## 8. 维护建议

1. 所有新增阶段性文档统一使用日期前缀。
2. 新文档开头必须明确“文档日期”或“最后更新日期”。
3. 若一个模块已经出现兼容层、demo 兜底或旧接口别名，文档必须写清楚“主路径”和“兼容路径”。
4. 不要同时维护两套互相冲突的说明文档口径。
5. 每次功能交付后，至少补一份 AI 交付报告和一份对应的总览或索引更新。

## 9. 结论

这次更新的关键不是“写更多文档”，而是“建立新的可信文档中心”。只要后续内容都围绕 `docs/项目总文档/` 和最新交付报告同步维护，项目的文档体系就能重新稳定下来。

# 项目阶段与里程碑重评估

文档日期：2026-05-17

## 1. 结论

AI-XsRead 当前不再适合使用“85% 完成度”“V1.0 Beta”“V1.1 / V1.5 / V2.0”这套 2025-10 的旧项目管理口径。

更准确的阶段定义是：

> 当前阶段：MVP+ 功能扩展后的产品化稳定期。

也可以表述为：

> 阅读体验增强与上线可信度收敛期。

这意味着项目已经不是“核心功能还没做完”的状态，而是主链路已成型、功能密度快速增加后，需要把体验、数据、验收、部署和文档统一收口。

## 2. 当前阶段拆解

### 产品阶段

项目已经从基础小说阅读平台推进到女性向阅读体验增强平台。首页、发现、搜索、详情、阅读、书架、个人中心、作者关注、上传、账号安全、管理后台等主链路已具备，后续重点是让这些链路更真实、更稳定、更有长期留存价值。

### 工程阶段

工程重点从功能补齐转入集成稳定化与技术债治理。当前已有大量新增能力，但也存在兼容入口、旧 mock、demo fallback、环境变量差异、数据库迁移边界、测试门禁和线上配置收口问题。

### 体验阶段

体验目标从“能读”转入“愿意长期读”。后续评估标准应围绕女性向重度读者是否愿意把它作为主用或副用阅读产品，而不是只看页面是否能打开。

### 管理阶段

项目管理方式应从旧报告驱动转为当前事实文档驱动。正式口径应以当前代码、`docs/项目总文档/`、`docs/AI完成后报告文档/` 和新版核心文档为准。

## 3. 已完成主能力

### 用户端

- 首页、发现、搜索、详情、阅读、书架、历史、个人中心
- 登录、注册、兴趣引导、上传小说、账号安全
- 关注作者、书签、划线、想法、段评气泡、成就
- 阅读设置、护眼/夜间、翻页、TTS、阅读进度同步

### 用户后端

- 认证、小说、章节、推荐、搜索、分类
- 书架、阅读进度、阅读历史、用户统计
- 评论、书签、划线、段评、作者关注、兴趣标签、成就
- TXT 导入、头像/图片上传、数据库迁移启动机制

### 管理端

- 管理员登录
- 数据看板
- 小说管理、章节管理、用户管理、评论管理
- 图片上传和管理员初始化脚本

## 4. 当前未完成的不是“功能清单”，而是“可信度清单”

旧报告习惯用“已完成 / 未完成 / 百分比”表达项目状态。当前更应该用可信度表达：

| 领域 | 当前判断 | 下一步 |
| --- | --- | --- |
| 阅读主链路 | 已成型 | 多断点真实验收 |
| 互动链路 | 已具备接口和前端入口 | 验证段评、划线、书签、评论在真实数据下稳定 |
| 个人中心 | 页面和入口已具备 | 统计数据真实化，避免假数字 |
| 管理后台 | 基础能力具备 | 补齐权限边界、测试和操作文档 |
| 数据库 | 多轮迁移已存在 | 明确初始化和增量迁移边界 |
| 文档 | 已完成核心迁移 | 继续清理旧入口和归档历史资料 |

## 5. 下一阶段里程碑

### M1：文档口径收敛

目标：所有当前入口文档都使用 2026-05-17 后的新口径。

验收：
- 旧文档已归档或删除
- 当前总览、接口、数据库、启动、快速指南均有日期化文档
- README 不再保留 Vue/Vite 模板口吻

### M2：真实验收矩阵建立

目标：将“功能看起来可用”升级为“真实场景可验收”。

验收：
- 建立登录态 fixture 与稳定测试数据
- 阅读页、详情页、书架、个人中心、段评、划线、TTS、关注作者进入 e2e 验收范围
- 至少覆盖主移动断点和桌面断点

### M3：数据真实性收口

目标：用户可见数字不得继续依赖假数据。

验收：
- 个人中心统计、详情页评分/人数、阅读历史、成就展示明确数据来源
- 无数据时展示空状态，不展示伪造繁荣

### M4：上线可信度检查

目标：部署、反代、上传资源、SPA fallback、环境变量和数据库迁移具备可重复检查清单。

验收：
- 用户端和管理端均有启动与部署检查项
- 上传资源和头像路径可追踪
- 公开 API 不被鉴权误拦
- 深链刷新不再 404

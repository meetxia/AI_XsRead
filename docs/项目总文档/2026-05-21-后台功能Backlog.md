## 2. 跨子工程治理项

### 2.1 admin_logs 与 code_admin_logs 双审计表整合

- **现状**：
  - `admin_logs`（旧）：novel/user/comment 在写
  - `code_admin_logs`（新）：code/contact/membershipAdmin 在写
- **影响**：查询管理员历史操作时需要 UNION 两张表。
- **建议**：
  - **方案 A**：保留双表，新加 SQL VIEW 统一对前端暴露
    ```sql
    CREATE VIEW v_admin_audit AS
    SELECT id, admin_id, NULL AS admin_username, action, module, target_id, description, ip, user_agent, created_at, 'admin_logs' AS source FROM admin_logs
    UNION ALL
    SELECT id, admin_id, NULL AS admin_username, action, NULL AS module, NULL AS target_id, COALESCE(target, '') AS description, ip, NULL AS user_agent, created_at, 'code_admin_logs' AS source FROM code_admin_logs;
    ```
    优点：不动既有写入代码。
  - **方案 B**：把 code_admin_logs 写入逻辑迁回 admin_logs（迁移期保留两份兼容）
    优点：长期清晰，缺点：影响面大。
- **建议先方案 A**，作为 backlog 项。

### 2.2 banners / announcements 表是否应该生效"defaults"

- 如果产品决定不做这两个功能，建议把这两个表的迁移注释掉（但**不要回滚**，因为 admin-backend 控制器随时可能加回）。

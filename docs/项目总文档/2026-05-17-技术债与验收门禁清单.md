# 技术债与验收门禁清单

文档日期：2026-05-17

## 1. 定位

本文用于承接旧系统优化报告中仍有效的工程建议，并结合当前项目真实状态重新排序。

当前技术债治理目标不是追求一次性重构，而是让项目进入可验收、可部署、可维护的稳定状态。

## 2. 技术债清单

### 兼容入口债

- `/reading/:id` 与 `/read/:id/:chapter?`
- `/api/auth/refresh` 与 `/api/auth/refresh-token`
- `/api/novels/categories` 与 `/api/categories`
- 前端 demo fallback 与真实 API 混用

处理原则：保留兼容入口时必须标明主入口；废弃入口必须先有迁移说明。

### 数据债

- 部分统计数据仍可能为 0、mock 或 demo fallback
- 阅读时长、连读天数、评分人数等口径需要统一
- `reading_progress` 与 `reading_history` 的展示态/埋点态边界需要明确

### 数据库迁移债

- 自动迁移脚本与手工 SQL 并存
- 旧 SQL 与当前 schema 可能冲突
- 部分迁移脚本命名不符合自动执行规则
- 管理端脚本与用户端迁移缺少统一索引

### 管理端债

- 权限粒度仍偏粗
- 部分页面占位或半成品
- 创建/删除接口返回码与前端拦截器存在潜在冲突
- 上传错误处理和 refresh token 链路需要复核

### 测试债

- 部分 e2e 仍受环境 gate 限制
- 后端集成测试依赖数据库环境
- 管理端缺少正式自动化测试
- 接口契约测试不足

### 部署债

- SPA fallback
- API 反代
- 上传资源路径
- 头像/封面路径标准化
- 环境变量差异
- 生产日志和调试开关

## 3. 验收门禁

### 文档门禁

- 新文档必须有日期
- 当前入口文档必须在 `docs/项目总文档/` 或对应正式目录中
- 历史文档必须明确标记归档
- README 不得保留模板口吻

### 功能门禁

- 主链路必须覆盖：首页 -> 详情 -> 阅读 -> 书架/历史 -> 个人中心
- 互动链路必须覆盖：评论、段评、划线、书签、关注作者
- 未登录和登录态都要验收

### 数据门禁

- 用户可见数字必须有数据来源
- 没有数据时必须显示空状态
- 不得继续使用无法解释的固定展示数字

### 端到端门禁

- 至少覆盖移动端和桌面端
- 核心阅读交互必须可自动化验证或手动验收记录
- e2e 环境依赖必须写入快速启动文档

### 部署门禁

- 深链刷新不 404
- `/api` 和 `/uploads` 反代正确
- 管理端 `/api/admin` 可用
- 数据库迁移状态可查看
- 上传资源可访问

## 4. 当前优先级

1. 真实验收矩阵
2. 数据真实性
3. 数据库迁移边界
4. 管理端权限和测试
5. 兼容入口清理
6. 性能监控和缓存策略
7. 长期架构升级

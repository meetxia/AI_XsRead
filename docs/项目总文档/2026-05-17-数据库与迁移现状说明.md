# MOMO小说（AI-XsRead）数据库与迁移现状说明

文档日期：2026-05-17

## 1. 当前数据库边界

项目当前围绕 MySQL 8.0 运行，数据库设计已经从早期“基础用户 + 小说 + 章节”扩展到“阅读行为、书架、评论、划线、段评、作者关注、兴趣标签、成就、统计”等多子系统。

## 2. 用户端后端数据库配置

配置入口：
- `backend/src/config/index.js`
- `backend/src/config/database.js`

核心特征：
- 强制校验数据库与 JWT 环境变量
- 默认端口 `3306`
- 连接池使用 `mysql2/promise`
- 字符集固定为 `utf8mb4`
- 时区固定为 `+08:00`
- 关闭多语句查询

## 4. 管理端数据库初始化

管理端后端仍保留独立初始化脚本与检查脚本：
- `admin-backend/scripts/check-all-tables.js`
- `admin-backend/scripts/check-and-create-tables.js`
- `admin-backend/scripts/check-database-data.js`
- `admin-backend/scripts/create-admin-tables.js`
- `admin-backend/scripts/init-admin.js`

说明：
- 管理端数据库与用户端数据库共享同一业务域，但初始化与管理表结构可能存在独立脚本。
- 文档中不要把管理端初始化脚本误写成用户端主库初始化脚本。

## 6. 数据口径与风险点

1. 老旧文档中的表结构、字段名、脚本文件路径已经不再可靠，尤其是 `docx/` 下的旧 SQL 说明。
2. 若新旧迁移脚本并存，文档应明确“初始化脚本”与“增量迁移脚本”的区别。
3. 部分控制器内存在统计聚合与字段兼容逻辑，文档不要把旧字段名直接写成唯一真相。
4. 头像路径标准化、段落哈希、评论表迁移、阅读历史字段修复，说明数据库已经历多轮演进，正式文档应保留“迁移背景”而不是只写最终结果。

## 7. 建议补强的数据库文档内容

后续建议在正式数据库设计文档中补上：

1. 当前实际表清单
2. 每张核心表的关键字段
3. 迁移脚本执行顺序
4. 初始化与增量更新的边界
5. 旧脚本是否仍应保留为回滚工具

## 8. 结论

当前数据库不是单一静态 schema，而是“持续演进中的业务型 MySQL 结构”。文档更新时要以当前代码中的配置、控制器与迁移文件为准，而不是以旧 SQL 备份为准。

## 1. 迁移机制概述

### 1.1 迁移流水线

- 入口：`backend/database/migrations/migrate.js`
- 触发：用户后端启动时自动执行（`backend/src/app.js` 在 `app.listen` 前 await `runMigrations()`），不需要单独命令
- 顺序：按文件名时间戳升序逐个执行
- 状态记录：`migrations` 表（自动创建），记录已执行版本与执行时间
- 健康检查上报：`/api/health` 返回 `migrations.latestVersion` / `total` / `lastAppliedAt`

### 1.2 命名规则（强制）

```
^\d{12}__描述.sql$
```

例：`202605200001__add_admin_tables.sql`

不符合此规则的 `.sql` 文件**不会**进入迁移流水线。这正是历史归档（`.archived` 后缀）应有的状态。

### 1.3 幂等要求

SQL 必须**幂等**——重复执行不会出错：

- 创建表：`CREATE TABLE IF NOT EXISTS`
- 插入种子：`INSERT ... ON DUPLICATE KEY UPDATE` 或 `WHERE NOT EXISTS`
- 加列 / 加索引：`migrate.js` 内置忽略 `ER_DUP_FIELDNAME` / `ER_DUP_KEYNAME` / `ER_TABLE_EXISTS_ERROR`，可放心写裸的 `ALTER TABLE ADD COLUMN`

### 1.4 不支持的语法

**不得使用 `DELIMITER //`**：`splitSqlStatements` 用 `;` 分隔，不识别此指令，会切碎存储过程定义。如需复杂逻辑，改用 `INSERT IGNORE` / `ON DUPLICATE KEY` 或在应用层处理。

### 1.5 legacy 归档原则

历史上未成功执行过的 SQL 文件（往往与当前 schema 不兼容，如外键类型不一致），必须保持 `.archived` 等扩展名归档，**不得**改名为时间戳让其进入迁移序列。重命名 legacy SQL 这件事 2026-05-18 已经导致一次 server boot 失败回归，参见 commit `036c15f`。

## 5. 常见迁移坑

### 5.1 主键类型不一致（已修复）

`user_preferences` 与 `notifications` 早期版本写成 `BIGINT UNSIGNED`，但 `users.id` 是 `INT UNSIGNED`。Sprint 1 / T-2.1 / T-2.2 已统一为 `INT UNSIGNED`（5/20 修订）。

**幂等性提示**：如果曾在某个开发机执行过旧版（`BIGINT`），需先 `DROP TABLE user_preferences; DROP TABLE notifications;` 再重跑迁移。生产环境从未执行过旧版，可直接跑新版。

### 5.2 DELIMITER 不支持

`migrate.js` 的 `splitSqlStatements` 不认识 `DELIMITER //`，会切碎存储过程。所有迁移**禁止**使用 DELIMITER。如需触发器或过程，放在 `triggers.sql` / `procedures.sql`，由运维手动跑。

### 5.3 改名 legacy 导致 server boot 失败（教训）

2026-05-18 commit `036c15f` 修复一次回归：上一次有人把 `legacy_*.sql.archived` 改名加上时间戳前缀，让它进入流水线，结果与 `comments` 当前 schema 外键类型不兼容，server 启动失败。

**纪律**：legacy SQL 永远保持 `.archived` 后缀，**不得**改名。

### 5.4 charset / collate 显式声明

所有迁移**必须**显式声明 `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci`。Sprint 1 已修订 user_preferences 与 notifications 两张表的 collate 缺失。

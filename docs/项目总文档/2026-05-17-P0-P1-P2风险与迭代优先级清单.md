# P0/P1/P2 风险与迭代优先级清单

文档日期：2026-05-17

## 1. 使用说明

本文替代 2025-10-27 旧项目经理报告中的“P0/P1/P2 优化建议”。旧报告的优先级建立在早期状态上，已经不能直接使用。

当前优先级判断基于：

- 当前代码实现
- 2026-05-17 新版核心文档
- 近期 AI 交付报告
- 项目经理复盘结论

## 2. P0：必须优先处理

### P0-1：正式口径混乱

风险：旧文档、旧 README、旧 SQL 说明、旧研究报告仍可能误导协作者，让人误以为项目仍处于 2025-10 的 Beta 阶段。

处理：
- 旧报告迁移到 `docs/历史研究归档/`
- 当前入口统一使用 `docs/项目总文档/`
- 所有新文档使用日期前缀

### P0-2：真实验收不足

风险：单元测试和 PBT 增强不等于真实用户链路已验收。部分 Playwright e2e 仍依赖环境 gate。

处理：
- 建立登录态 fixture
- 建立稳定测试数据
- 阅读、详情、书架、段评、划线、TTS、关注作者、成就进入 e2e 验收矩阵

### P0-3：线上可信度

风险：SPA fallback、API 反代、上传资源、头像路径、公开 API 鉴权边界都曾暴露问题。

处理：
- 建立上线检查清单
- 固定 Nginx / 反代配置文档
- 检查 `/uploads`、头像、封面和深链刷新

### P0-4：数据真实性

风险：部分统计、评分、阅读时长、连读天数、人数等展示可能仍存在硬编码、全 0 或 demo fallback。

处理：
- 所有用户可见数字分为真实接口、空状态、不展示三类
- 禁止继续保留“看起来很漂亮但不真实”的运营数字

### P0-5：数据库迁移边界

风险：初始化脚本、自动迁移、手工补丁、历史 SQL 并存，容易造成开发库、测试库、线上库 schema 不一致。

处理：
- 明确 `backend/database/migrations/` 自动迁移规则
- 明确 `docs/06-数据库脚本/` 只作为脚本资料和手工参考
- 建立迁移执行记录和回滚策略

## 3. P1：近期推进

### P1-1：阅读页产品化验收

范围：
- 360 / 412 / 768 / 1024 / 1440 断点
- 翻页模式切换
- TTS voices 加载
- 段评气泡点击命中
- 划线工具菜单遮挡
- 阅读进度跨设备恢复

### P1-2：个人中心真实化

范围：
- 用户统计
- 阅读历史
- 成就
- 关注作者
- 书签、划线、想法聚合

目标：让“我的”页面从展示型页面变成真实数据聚合页。

### P1-3：发现与推荐闭环

范围：
- 冷启动推荐
- 兴趣标签
- 阅读时长
- 书架行为
- 同类作品推荐
- 作者其他作品

目标：先做可解释推荐和运营榜单，再考虑更重的推荐基础设施。

### P1-4：管理后台文档与权限边界

范围：
- 管理后台现状总览
- 操作流程
- 权限边界
- 内容管理流程
- 返回码与前端拦截器一致性

### P1-5：CI 与测试门禁

范围：
- 后端集成测试稳定数据库
- 核心 e2e 加入可重复执行流程
- 覆盖率门禁逐步恢复
- 接口契约测试

## 4. P2：中长期方向

### P2-1：内容规模与数据质量

目标不是简单堆数量，而是建立导入、清洗、分类、标签、封面、章节质量的内容运营工程。

### P2-2：移动端 / PWA / 离线阅读

适合作为阅读链路稳定后的增强方向。

### P2-3：作者平台和商业化

包含作者后台、VIP、打赏、付费章节、版权保护等，但前置条件是内容规模、留存数据和运营后台成熟。

### P2-4：AI 探索

AI 摘要、智能推荐解释、AI 辅助书评等可以进入探索池，但不应压过当前阅读体验和上线可信度。

### P2-5：技术架构长期升级

TypeScript、微服务、DevOps 深化、监控体系可进入长期技术路线，但不应优先于 P0/P1。

# 用户体验对标微信读书 / 晋江 — 优化计划清单

> 文档日期：2026-05-17
> 作者：老书虫体验视角 + 源码审计
> 对标产品：微信读书、晋江文学城

## 一、规划目标

把"MOMO小说"从"能读 TXT"拉到"老书虫不愿离开"。判断一条任务是否进入计划，标准只有一条：

> **如果不做，已经习惯微信读书 / 晋江的女性向重度用户，不会愿意把它当成主用阅读 App。**

整体规划期：**4 周** + 后续优化窗口。

---

## 二、当前最大问题（一句话总结）

1. **阅读页是"占位 UI"**：书签 / 加书架 / 收藏 / 推荐 / 分享按钮全部没绑事件；翻页只能上下滑；进度同步只到章级。
2. **详情页和个人中心大量"假数据"**：评分人数、3,421 人评分、12.4 万在读、阅读时长 287h、连读 14 天 — 全部硬编码。
3. **核心差异化能力缺失**：划线 / 想法 / 段评 / 听书 / 分榜 / 关注作者更新红点 — 一个都没有。
4. **后端能力没被前端消费**：成就规则、热搜、收藏、点赞、章节抽屉组件、阅读设置 composable — 已经写好，前端没接。

## 十一、最关键的一条建议

如果只能选 **一个月内做的最关键单项**，我会选：

> **P1-B-1 划线 + P1-B-2 想法 + P1-B-3 段评（三件套一起做）**

理由：
- 这是微信读书让用户搬不走的根本原因——一年的笔记 / 想法都在那。
- 这同时也是晋江老粉黏性最高的功能（"段评比正文还好看"）。
- 你现有数据模型只需多加 2 张表（`highlights` / `paragraph_comments`），改动收敛。
- 用户感知最强、迁移成本壁垒最高。

完成 P0 + P1-B-1/2/3 后，可以邀请 5-10 个晋江老粉做闭门测试，听她们说"这功能比晋江段评好用"，就成了。

---

## 十二、风险点

1. **段落锚点稳定性**：作者后期修文导致段索引漂移 → 建议章节正文存储时同时保存 `paragraph_hash`（段落首 50 字 hash），段索引漂移时按 hash 找回。
2. **运行时建表的技术债**：`commentController.ensureCommentTables` 应迁到 migrations 目录，启动时一次 migrate。否则与 `init_step1.sql` 里的旧 `comments` schema 冲突。
3. **冷启动数据稀疏**：当前只有 35 部小说，推荐算法升级前要先扩充内容。建议同步推动管理后台批量上传 + 数据清洗工具。
4. **测试覆盖空白**：以上功能直接上线没有回归保障。建议至少给"阅读进度同步""划线""段评"三个写 e2e（Playwright）。

### 3. 定时任务启动（手动补完）✅

修改 [backend/src/app.js:115-130](backend/src/app.js#L115-L130) 的 `startServer` 函数，在数据库迁移完成后、`app.listen` 之前注入：

```js
if (process.env.ENABLE_SCHEDULED_JOBS !== 'false') {
  try {
    const { scheduledJobs } = require('../jobs/scheduledJobs');
    if (scheduledJobs && typeof scheduledJobs.start === 'function') {
      scheduledJobs.start();
      app.locals.scheduledJobs = scheduledJobs;
      console.log('⏰ 定时任务已启动');
    }
  } catch (jobErr) {
    console.warn('⚠️  定时任务启动失败（不影响主服务）:', jobErr.message);
  }
}
```

要点：
- `scheduledJobs.js` 真实导出是 `module.exports = { ScheduledJobs, scheduledJobs }`（class + 单例），所以解构 `scheduledJobs` 单例后调用 `.start()`
- `try/catch` 包裹，定时任务失败不阻塞主服务
- 通过环境变量 `ENABLE_SCHEDULED_JOBS=false` 可关闭（CI 中默认关）
- 测试场景下 `require.main !== module`，不会重复启动

**验证：** `node -c backend/src/app.js` 通过。

## 四、已知问题

1. **patch_v2 整合迁移的执行风险**
   - 新迁移使用 `IF NOT EXISTS`/`ON DUPLICATE KEY UPDATE` 兜底，但 MySQL 8 对 `ALTER TABLE ADD COLUMN IF NOT EXISTS` 的支持仅在某些发行版可用
   - 推荐在 staging 环境跑一次确认，再上线
   - 子代理执行时记录了部分语句若不支持 `IF NOT EXISTS` 会用 information_schema 探测包装，但实际兼容性需在真库验证

2. **scheduledJobs 启动后未做健康检查**
   - 当前只是 `try/catch` 启动，没有探针确认每个 cron 任务在跑
   - 后续建议接 `/api/health/jobs` 端点暴露 `scheduledJobs.tasks` 状态

3. **CI 未启用 E2E 测试**
   - `tests/e2e/*.spec.ts` 仍被 `E2E_READY !== '1'` gate skip
   - 启用方案：在 ci.yml 中添加专门的 e2e job，先启动 backend + frontend，再设 `E2E_READY=1` 跑 playwright
   - 暂未做，因 e2e 测试本身依赖登录态 fixture（属于 Sprint 3 的 P1-5 范围）

4. **cacheManager 实际接入接口数量**
   - 子代理只完成了 `getNovelDetail` 的接入；`getNovelList`/`getRecommendNovels` 等其他热点接口未改
   - 这是为了降低风险（现有 `memoryCache` 还在工作），后续可逐个迁移

5. **子代理两次中断**
   - 第一次：API socket 异常中断（仅跑 2 个 tool）
   - 第二次：执行到 25 个 tool 时停在 novelController 缓存接入的思考过程中
   - 影响：CI yaml 与 app.js 启动钩子是项目经理 AI 手动补完，**不是代理产物**
   - 同时 Track D 的报告（本文件）也是手动写就，建议团队 review 是否符合预期

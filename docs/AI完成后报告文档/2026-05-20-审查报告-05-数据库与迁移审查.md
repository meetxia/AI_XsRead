## 1. 总体结论

**评分：72 / 100（"骨架很完整，治理细节有缺漏"）**

数据库层是这个项目的工程亮点之一。迁移引擎健全（事务包裹、幂等错误吞掉、迁移记录表）；命名规范统一（`^\d{12}__描述.sql$`）；表结构完成度高（22 张业务表 + 1 张 migrations 元表）；新增功能（个人中心 MVP、激活码、关注作者、段评、划线）的 SQL 与对应控制器字段交叉一致。

但发布前需要警惕几条治理漏洞：
1. **管理端表（admin_users/admin_logs/banners/announcements/statistics_daily）走的是独立 `create-admin-tables.js`**，**没有进入 migrate 流水线**——这意味着 deploy.py 启动时 `runPendingMigrations(pool)` 不会创建管理员表，迁移序列对生产环境的"管理后台首次部署"无效，必须人手跑 `node scripts/create-admin-tables.js`。
2. **几乎所有表都没有外键约束**，引用完整性由应用层保证；这是项目主动选择（避免 DDL 锁），但任何越权 / 错位写入都会留下孤儿数据。
3. **新增个人中心两张表（`user_preferences` / `notifications`）字段类型为 `BIGINT UNSIGNED`，但既有用户表是 `INT UNSIGNED`**——一处类型不一致，未来若加外键会失败。

整体可部署，治理项继续推进即可。

## 4. 迁移引擎评估（`backend/database/migrate.js` 全文 238 行）

### 4.1 命名校验

- `MIGRATION_PATTERN = /^(\d{12})__(.+)\.sql$/`（line 5）
- `listMigrationFiles()` 严格 `.filter(filename => MIGRATION_PATTERN.test(filename))`（line 91）
- 不匹配的文件（如 `legacy_*.archived`）**自然被跳过**，归档机制有效。✅

### 4.2 splitSqlStatements（line 21-83）

- 自实现 SQL 切分：处理单引号 / 双引号 / 反引号 + `--` 行注释 + `#` 行注释 + `/* */` 块注释
- **不处理 `DELIMITER //`**——AGENTS.md 已显式禁用，与代码一致 ✅
- **不处理多行字符串中转义**：但 backtick / quote 切换正常
- 测试覆盖：`backend/tests` 目录下应有针对 splitSqlStatements 的单元测试（待报告 02 核实）

### 4.3 幂等吞错

```js
const BENIGN_IDEMPOTENCY_ERRORS = new Set([
  'ER_DUP_FIELDNAME',
  'ER_DUP_KEYNAME',
  'ER_TABLE_EXISTS_ERROR'
]);
```
- `executeStatement()` 命中后 `console.warn` 跳过（line 132-137）
- 这恰好覆盖了 `patch_v2_consolidation.sql:18-19` 那种"既存字段反复 ALTER"的幂等需求 ✅
- 没覆盖：`ER_DUP_ENTRY`（INSERT 撞 UK）、`ER_BAD_FIELD_ERROR`（字段不存在的 UPDATE）。前者通过 `ON DUPLICATE KEY UPDATE` 在 SQL 内自处理，后者会真失败——这是合理设计。

### 4.4 事务与回滚

- `runSingleMigration()` 在 `connection.beginTransaction` 可用时全程包裹（line 156-184）
- 单个迁移失败 → rollback + 抛出 `MigrationError`
- **MySQL DDL 不能回滚**（这是 MySQL 限制，非脚本 bug），但 INSERT/UPDATE 部分能回滚
- 失败后 `app.js:158-161` 会 `process.exit(1)`，PM2 重启循环——这就是为什么 AGENTS.md 强调"legacy SQL 不得改名"

### 4.5 迁移记录表

- `ensureMigrationsTable` 在 `runPendingMigrations` 第一步建表（line 113-121）✅
- `loadAppliedMigrations` 排序读取（line 123-126）
- 记录 INSERT 在 transaction 内（line 168-171），原子性 ✅
- 健康检查 `app.js:115-118` + `routes/index.js:32-49` 暴露 `latestVersion / total / lastAppliedAt` ✅

### 4.6 迁移引擎结论

整体评估：**A- 等级**。业内中小团队迁移引擎能写到这种水平的不多。可改进项：
- 增加 `down` migration 支持（目前只能 forward）
- 增加 `ER_DUP_ENTRY` 容忍开关
- 给 `splitSqlStatements` 写明 `DELIMITER` 不支持的 throw 提示

## 6. 数据完整性与外键现状

**全项目唯一的外键约束**：`activation_codes.batch_id → code_batches(id) ON DELETE RESTRICT`（`202605171103:26`）。

其他所有 user_id / novel_id / chapter_id 关联**全部通过应用层维护**。

### 影响

- **优势**：DDL 可以更快（加字段不锁外键检查）；删除用户/小说不会被外键阻断。
- **风险**：
  - 删除用户后，`bookshelf` / `reading_progress` / `comments` / `paragraph_comments` / `highlights` / `notifications` / `user_preferences` 都会留**孤儿数据**。
  - 没有 `ON DELETE CASCADE`，需要应用层显式清理（待报告 02 核实 `userController.js` 是否有删除用户的级联清理）。
  - 一旦未来加外键，`BIGINT UNSIGNED` vs `INT UNSIGNED` 会先撞类型不匹配错（见 5.1 / 5.2）。

### 推荐治理方向

不要全表加 FK（DDL 开销大），但应：
1. 类型对齐：把 user_preferences / notifications 的 user_id 改回 INT UNSIGNED。
2. 关键路径加约束：activation_codes 已经有 FK 是好榜样，user_achievements / code_redemption_logs 也可加。
3. 应用层注册"删除用户钩子"统一清孤儿。

## 7. 命名 / 时间戳 / 字符集一致性

### 7.1 时间戳

| 字段名 | 出现表 | 类型 | 默认 |
|---|---|---|---|
| `created_at` | 全部业务表 ✅ | DATETIME | DEFAULT CURRENT_TIMESTAMP |
| `updated_at` | 大部分表 | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP |
| `last_read_time` / `last_login_time` / `last_update_time` / `read_time` / `publish_time` / `added_time` / `unlocked_at` / `read_at` / `void_at` / `used_at` | 散见 | DATETIME | NULL 或 CURRENT_TIMESTAMP |

**风险**：
- `bookshelf.added_time` ≠ 主流的 `created_at`，是历史遗留命名。
- `reading_history.read_time` ≠ `reading_history.created_at`——同一个语义两个不同字段名（`init_step1.sql:164` vs 一般约定）。
- `user_preferences` 只有 `updated_at`，**没有 created_at**——审计盲点。

### 7.2 字符集

- 业务侧 21 张表：`DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci` ✅
- 新增 user_preferences / notifications：`DEFAULT CHARSET=utf8mb4`（无 collate）❌
- 管理端 admin_users / admin_logs / banners / announcements / statistics_daily：`utf8mb4_unicode_ci` ✅
- migrations 元表：`utf8mb4_unicode_ci` ✅

### 7.3 主键类型

- 大部分：`INT UNSIGNED AUTO_INCREMENT`
- 个人中心 MVP：`BIGINT UNSIGNED`
- `system_contact`：`TINYINT UNSIGNED DEFAULT 1`（强制单行表，设计合理）

## 8. 索引与性能要点

### 8.1 设计良好的索引

- `novels` 多维度索引覆盖了首页排行 / 推荐 / 热门 / 最新更新 4 类查询场景，且有 FULLTEXT(title, author, description) 给搜索（`init_step1.sql:67-75`）。✅
- `comments` 软删除字段 `deleted_at` 有独立索引——避免 `WHERE deleted_at IS NULL` 全表扫描。✅
- `notifications.idx_user_unread (user_id, is_read, created_at)` 完美对齐控制器 `WHERE user_id = ? AND is_read = 0 ORDER BY created_at`。✅
- `bookshelf` 有 4 个索引覆盖了"按用户取最近阅读 / 按类型筛选 / 唯一约束"。✅
- `paragraph_comments.idx_novel_chapter_paragraph (novel_id, chapter_id, paragraph_index)` 是前端段评气泡的核心查询路径。✅

### 8.2 可能冗余 / 可去重

- `users` 有 `idx_username` + UK 隐式索引：UK 已经是索引，`INDEX idx_username` 重复（`init_step1.sql:7,21`）。可删 INDEX 节省写开销。
- `users` 同样 `idx_email` 与 UK 重复。
- `categories.name` UK + `init_step1.sql` 没看到 `INDEX idx_name`——这里 OK。
- `tags.name` UK + `INDEX idx_name`——又重复（line 102, 105）。

### 8.3 缺失 / 待评估

- `comments.novel_id` 列出了，但常见查询 `WHERE novel_id = ? AND parent_id IS NULL ORDER BY created_at DESC` 需要复合索引 `(novel_id, parent_id, created_at)` 才能 ICP 优化——目前只有单列索引。
- `reading_progress.idx_novel_id` 在生产基本不会用——大部分查询是按 user_id 取我读到哪——`uk_user_novel` 已覆盖，可考虑删除单 novel_id 索引。
- `bookshelf.idx_novel_id` 同样基本无用，主路径都是 user_id 起步。

### 8.4 大表风险

- `chapters.content LONGTEXT` + `paragraph_hashes JSON`——字数大的小说一行可能很大；导入 35 部 TXT 时若不分章节存，性能堪忧（待报告 02 核实导入脚本）。
- `reading_history` 没有归档机制，长期会膨胀，热点查询 `WHERE user_id ORDER BY read_time DESC LIMIT 20` 索引在，但表越来越大 INSERT 成本会上升。

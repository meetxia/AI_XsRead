## 0. 一句话结论

| 系统 | 完善度 | 关键阻塞 |
|---|---|---|
| 用户系统 | 🟡 75% | 无密码找回、无邮箱/手机验证、SecurityPage 多个子项后端未对接 |
| 会员系统 | 🟡 70% | 无线上支付、无到期提醒、无差异化会员权益、无续费推送 |
| 后台激活码系统 | 🟡 65% | **管理前端页面缺失（运营无法独立操作）**、无支付闭环、无失效时间、无统计面板 |
| **端到端业务闭环** | 🚧 60% | 用户购买 → 激活 → 续费提醒 → 续费 整链路不闭环 |

底线：**功能链路（注册 → 登录 → 激活 → 阅读 VIP）跑得通，但运营自助 / 商业闭环不闭合**。当前依赖"管理员手工生码 + 客服微信售卖 + 用户手工输入"的极简运营模式，可以小规模上线，但难以规模化。

### 1.2 存在的缺口 🟡 ❌

| # | 缺口 | 严重度 | 说明 |
|---|---|---|---|
| U-1 | **无密码找回 / 重置流程** | P0 | grep `forgot/reset-password/sendCode` 全仓零命中。一旦用户忘记密码必须联系客服手工重置，且后台 `init-admin.js` 也只支持初始化管理员密码，缺少用户级别的"邮箱重置链接 / 短信验证码"机制 |

| U-3 | **无图形/滑块验证码 / 设备指纹** | P1 | 仅靠 IP 限流（10 次/15min）防爆破。代理池场景下基本失效。注册接口没有"防机器人"层 |
| U-4 | **logout 仅前端清 token，无服务端黑名单** | P2 | [authController.js:304](../../backend/src/controllers/authController.js#L304) 只返回 success；access token 在 7 天内若被截获仍然有效。建议引入 Redis token 黑名单或缩短 access token 时长 |
| U-5 | **change-password 未挂限流** | P2 | [routes/auth.js:37](../../backend/src/routes/auth.js#L37) 无 rate limit，登录后高频调用可暴力试旧密码（需先登录，风险次级） |
| U-6 | **SecurityPage 多项功能后端未对接** | P1 | 「绑定手机/邮箱、设备管理、注销账号」UI 已就位但后端无接口（[功能矩阵 #13](../项目总文档/2026-05-21-功能能力矩阵.md)）。GDPR / 个人信息保护法的"账号删除权"未实现 |
| U-7 | **无第三方登录** | P2 | LoginPage 有"三方按钮 UI 占位"但 backend/admin-backend 都没有 OAuth 路由 |
| U-8 | **admin 登录无失败次数锁定** | P1 | [审查报告 04 / Backlog #6](../项目总文档/2026-05-21-后台功能Backlog.md)。爆破成本仅取决于全局 rate limit |
| U-9 | **refresh 端点未挂限流** | P2 | [routes/auth.js:28](../../backend/src/routes/auth.js#L28) 没有 authLimiter，攻击面虽小但建议统一处理 |

### 1.3 推荐优先级（按业务影响）

1. **P0** U-1 密码找回流程（可先做"邮箱链接 + Redis 30min token"最小实现）
2. **P1** U-2 邮箱真实性 + U-3 图形验证码 + U-6 注销账号 + U-8 admin 失败锁定（这四项是合规与安全门槛）
3. **P2** U-4 / U-5 / U-9 限流与 token 黑名单收口
4. **P2** U-7 第三方登录（运营拉新效率，非阻塞）

## 4. 端到端流程审查

### 4.1 现状链路图

```
[管理员] ─POST /api/admin/codes/batches──► 后端生码 ──► CSV 导出
                                            │
                                            ▼
                                    code_batches + activation_codes（AES-GCM 加密入库）
                                            │
[运营/客服] ◄─手工拷贝明文码──────────────────┘
   │
   │ 微信/QQ 把码发给用户（依赖 system_contact 联系方式）
   ▼
[用户] ─输入 16 位码 ──► POST /api/user/membership/activate
                          │
                          ├─► FOR UPDATE 锁码
                          ├─► computeNewMembership 续期
                          ├─► UPDATE users.vip_*
                          ├─► UPDATE codes.status='used'
                          ├─► INSERT code_redemption_logs
                          └─► COMMIT
                          │
                          ▼
                    用户继续阅读 VIP 书 ──isMember()──► 通过
```

### 4.2 链路评估（每一步是否可独立运营）

| 步骤 | 后端能力 | 前端能力 | 运营自助度 |
|---|---|---|---|
| ① 管理员生码 | ✅ | ❌ 无页面 | **0%（必须工程师 curl）** |
| ② 批次导出 CSV | ✅ | ❌ 无页面 | 0% |
| ③ 客服把码给用户 | — | — | 100%（人肉微信） |
| ④ 用户激活 | ✅ | ✅ | 100% |
| ⑤ 用户阅读 VIP 内容 | ✅ | ✅ | 100% |
| ⑥ 到期前提醒续费 | ❌ | ❌ | 0% |
| ⑦ 用户找回密码 | ❌ | ❌ | 0% |
| ⑧ 售后停用会员 | ✅ | ❌（用户详情页有按钮但 UI 简陋） | 50% |
| ⑨ 兑换日志查询 | ✅ | ❌ 无页面 | 0% |
| ⑩ 续费 / 升级 | 走 ① 重来 | — | 0% |

### 4.3 关键断点

1. **[最严重]** ①②⑨ 共同断点：**管理前端激活码页面缺失**，整个生码-售卖-审计链路必须依赖工程师介入
2. **[严重]** ③ 客服售卖：依赖 system_contact 单点，且无订单凭证、无对账、无支付通道
3. **[严重]** ⑥ 续费提醒：通知中心已就位，但没有续费提醒 cron 任务驱动
4. **[严重]** ⑦ 密码找回：完全缺失，是合规与用户体验的硬门槛
5. **[一般]** ⑩ 续费体验：用户必须重新经历"找客服 → 拿码 → 输入码"三步，没有"一键续费"

## 7. 给业务的关键问题（需要决策）

在做下一步迭代前，建议项目负责人在以下三个问题上做选择：

### Q1：商业模式 — 是继续"客服微信售卖" 还是 "线上自助支付"？
- 选 A 客服模式：**必须先做管理前端激活码页面（C-1）**，否则运营继续依赖工程师
- 选 B 自助支付：**直接接入微信小程序支付/支付宝 H5（M-1）**，订单表 + 凭证 + 对账一并做
- **建议**：B 优先；A 作为兜底渠道保留

### Q2：差异化会员权益 — 三档套餐功能上是否要拉开差距？
- 当前三档功能完全相同，只靠"时长"撑价格阶梯
- 如果不拉开差距，建议**永久卡定价上限敏感**（用户算账后会发现年卡 ×3 ≈ 永久卡反而更划算）
- 拉开差距需要：每日下载次数 / 同时在线设备数 / 听书时长 / 字体包等 quota 设计

### Q3：合规性优先级 — 密码找回 + 账号注销 是否纳入本 Sprint？
- 没有"账号注销"在国内属于**违反《个人信息保护法》第 47 条**（应用商店审核也会卡）
- 没有"密码找回"会显著抬高客服工单量
- 建议两项**纳入下个 Sprint 必交付清单**

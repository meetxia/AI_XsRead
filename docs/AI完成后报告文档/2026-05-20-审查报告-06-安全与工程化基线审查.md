## 1. 总体结论

**评分：63 / 100（"工程化已成型，安全有可执行风险"）**

工程化基线已经初具规模——`backend/scripts/deploy.py` 是一份认真写过的部署脚本（rsync `--checksum`、`.env` 排除、PM2 reload、健康检查、缓存对账），`ecosystem.config.js` 与 `nginx.conf` 与 `deploy-guide.md` 在路径口径上一致，`.github/workflows/ci.yml` 覆盖了 4 个子工程的 build/test/语法检查。但具体到生产可发布层面，仍有几处可执行的严重风险：

1. **CI 几乎不阻断**：4 步关键检测里 4 处使用了 `continue-on-error: true`，绿牌也可能是假绿牌。
2. **认证中间件每次请求打印 token / username / userId**：`backend/src/middlewares/auth.js:13-49` 把 Bearer token 前 20 字符与解析出的用户名写到 stdout，PM2 出文件 `../logs/backend-out.log` 永久保存。这是 OWASP A09 Logging Failures 风险。
3. **管理后端使用同一份 JWT secret 同时签 access 和 refresh**：`admin-backend/src/utils/jwt.js:11-23` 没有用独立的 `JWT_REFRESH_SECRET`，与用户后端的双密钥设计不一致，且 access+refresh 同密钥会让 refresh 泄露后等价于无限期 access。

工程化的"骨架"在，发布前需要把这几个口子收住。

## 2. 部署与运维基线

### 2.1 deploy.py（唯一受支持部署脚本）

- 文件：`backend/scripts/deploy.py`（1219 行）
- 主要能力（已读源码确认）：
  - rsync 增量上传 + 本地 sha256 manifest 对账（`_build_manifest` / `_summarize_manifest_changes`）
  - 排除 `.env`、`node_modules`、`tests`、日志、缓存
  - 对前端 dist、用户后端、管理前端 dist、管理后端分别有独立目标路径常量
  - PM2 reload，失败 fallback 为 start
  - 健康检查 `/api/health`，端口 8005 / 8001
  - 默认仅允许部署到 `/www/wwwroot/`（白名单根，避免误伤）
  - 支持 `--dry-run` / `--watch`
- 评估：**质量在线**。脚本本身写得审慎。
- 风险点：
  - **rsync 没有显式的 `--delete-excluded` 或显式 delete 策略说明**，需要对照实际命令行参数（脚本超长本次未看到 `_run_rsync` 完整段落，主代理保留追阅项）。
  - 脚本日志直写 `deploy.log` 在仓库内 `backend/scripts/deploy.log`（未在 `.gitignore`），多次部署会污染工作区。
- 状态：基本完成，需要收尾。

### 2.2 PM2 配置 `ecosystem.config.js`

| 进程 | 路径 | 端口 | 单进程 | 自动重启 | 内存上限 | 日志 |
|---|---|---|---|---|---|---|
| `xsread-backend` | `/www/wwwroot/xs.momofx.cn/backend` | 8005 | `instances: 1` | ✅ | 500M | `../logs/backend-{error,out}.log` |
| `xsread-admin-backend` | `/www/wwwroot/xs.momofx.cn/admin-backend` | 8001 | `instances: 1` | ✅ | 300M | `../logs/admin-{error,out}.log` |

  - **`watch: false`** 是正确的（生产应该禁用）。
  - **没有 `kill_timeout` / `listen_timeout`**：滚动 reload 无优雅关闭窗口。
- 状态：**基本完成需要收尾**。

### 2.3 Nginx `nginx.conf`

- 文件：`nginx.conf`（109 行）
- 已配项：
  - `gzip` ✅（含 svg、woff2，level 5）
  - 静态资源 365d 缓存 ✅
  - HTML 不缓存 ✅
  - SPA fallback `try_files $uri $uri/ /index.html` ✅
  - `/api` 反代到 8005 ✅
- **关键缺失**：
  - **整个文件只 listen 80**，没有 443 / SSL 配置。生产域名 xs.momofx.cn 不能裸 80 跑（`nginx.conf:5-6`）。
  - **没有强制 HTTPS 跳转**。
  - **没有 `admin.momofx.cn` 或 `/admin/` server block**——`AGENTS.md:209` 明确说"管理前端 dist 上传到服务器后默认无访问入口"，这与 `nginx.conf` 当前状态完全一致，意味着管理后台目前**不可访问**。
  - **没有 brotli**（虽然 vite 构建有 `vite-plugin-compression` 生成 .br，但 nginx 没启 `brotli on`，等于浪费）。
  - **CORS 通配符 `*`**（`nginx.conf:47`）+ 后端 cors 同时开 `Access-Control-Allow-Origin`：双层重复，且 nginx 这层是 wildcard 任意域，等于全开。建议要么在 nginx 删掉 CORS 头由后端统一处理，要么在 nginx 写白名单。
  - **没有安全响应头**：缺 `Strict-Transport-Security`、`X-Content-Type-Options nosniff`、`X-Frame-Options`、`Referrer-Policy`、`Permissions-Policy`、CSP。
  - **WebP 协商不完整**：`location ~* \.(jpg|jpeg|png)$` 只加了 `Vary: Accept`，没有 `map $http_accept $webp_suffix` 实际重写到 .webp 文件。
- 状态：**半成品**。

## 6. 依赖安全审计

### 6.1 主要版本表

| 子工程 | Express | helmet | rate-limit | bcrypt | mysql2 | 备注 |
|---|---|---|---|---|---|---|
| backend | 4.18.2 | 7.1.0 | 7.1.5 | bcryptjs 2.4.3 | 3.6.0 | 主体在 Express 4 |
| admin-backend | **5.1.0** | **8.1.0** | **8.1.0** | bcryptjs **3.0.2** | 3.15.3 | 主体在 Express 5 |
| ai-xsread-vue3 | — | — | — | — | — | Vue 3.5.22, axios 1.12.2, Vite 7.1.7 |
| admin-frontend | — | — | — | — | — | Vue 3.5.22, Element Plus 2.11.5, ECharts 6.0.0 |

**关键发现：用户后端是 Express 4 + helmet 7 + bcryptjs 2，管理后端是 Express 5 + helmet 8 + bcryptjs 3。**
- 不同主版本的中间件签名不一致（Express 5 默认 promise 错误处理）。
- Express 4 已经进入维护模式，长期需迁移到 5。
- bcryptjs 2.x 与 3.x API 完全兼容，但版本分裂会让两个 utils 没法共用。
- 同一仓库两个不同主版本，**这是技术债，但短期可接受**。

### 6.2 lockfile 状态

- 4 个子工程均存在 `package-lock.json`（已确认 `backend/package-lock.json`、`ai-xsread-vue3/package-lock.json`、`admin-backend/package-lock.json`、`admin-frontend/package-lock.json` 通过 ls 推断 — 待 git 跟踪状态由"6.3"小节追阅）。
- 工作区 `git status` 中 `backend/scripts/.deploy-cache.json` 被修改且追踪——这个文件应当被 .gitignore 忽略（已经在 line 35 排除了，证明历史上确实写过缓存）。

### 6.3 其他依赖观察

- **CSRF 在用户后端启用**：`csurf ^1.11.0`（`backend/package.json:26`），但 `csurf` 包早被 npm 标 deprecated（自 2022）。建议迁移到 `csrf-csrf`。
- **swagger-jsdoc + swagger-ui-express** 在生产是否暴露需检查：`backend/src/app.js:80-89` 无任何鉴权地挂载在 `/api-docs`，**生产环境也会暴露完整 API 文档**。这是 Info → Medium 风险。
- **redis vs ioredis** 同时存在：`backend/package.json:34,38`，二选一即可。
- **multer 1.x → 2.x**：用户后端 1.4.5-lts.1，管理后端 2.0.2，又是分裂。

### 7.4 限流

- 用户后端：`backend/src/app.js:45-67`

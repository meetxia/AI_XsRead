## 4. 跨模块"金线问题"（必须发布前修）

下面按"链路风险"而非单模块视角列出。这些是会**互相串成攻击链**的硬控制项。

### 金线 1 — 管理后台默认密码 + SQL 注入 + 限流失效

**链路**：
1. `admin-backend/scripts/init-admin.js:73-91` 默认 `admin/admin123`，不强制改。
2. `admin-frontend/src/views/Login/index.vue:52,72-74` reactive 默认填充 admin/admin123 + 文案明示。
3. `admin-backend/src/app.js` 缺 `app.set('trust proxy', 1)` → nginx 反代后所有 IP = 127.0.0.1 → 全站共享 5/15min 登录限流（详见报告 04 §4.2）。
4. `admin-backend/src/controllers/novelController.js:19,59` `ORDER BY n.${sort}` 直接拼接 query 参数（SQL 注入活靶子）。
5. `nginx.conf` 没有 admin.momofx.cn server block → 但攻击者可以从用户域同源 cookie 攻击 / nginx 不强制 HTTPS → 中间人也可。

**结论**：任何在生产部署后未改默认密码的实例 = 公开数据库读权限。**这是 P0-Critical。**

### 金线 2 — 构建闸门失效

**链路**：
1. `admin-frontend/src/views/Settings/index.vue:272,369` ASI bug 让 `npm run build` 退出码 1。
2. CI 中 admin-frontend job **没有 `continue-on-error`**，所以 PR 应该被卡住——但同时项目也没启用分支保护规则强制 CI（待复核 GitHub branch protection 设置）。
3. 用户后端 `npm test` 失败 2 用例（fontSize / novelStatus）；CI 中 `Jest tests` 显式 `continue-on-error: true`，**测试失败已经在主分支被放行了**。
4. 部署链路 `deploy.py` 真增量 + PM2 reload，但**不会自动跑 CI**——deploy 自身没卡门。

**结论**：当前主分支不可信构建状态，任何 PR 合并到 main 后人手 `deploy.py --all` 也能上线。**P0-Critical。**

### 金线 3 — 认证与日志泄漏

**链路**：
1. `backend/src/middlewares/auth.js:13-49` 用户后端认证中间件每请求 console.log token 前 20 字符 + userId/username。
2. PM2 写入 `logs/backend-out.log` 永久保存。
3. `backend/src/app.js:80-89` `/api-docs` 在 production 也无鉴权暴露 swagger。
4. `nginx.conf` 没 HSTS / CSP，token 会在中间人场景暴露。

**结论**：单条 token 可以被 SSH 上服务器或拿到日志的运维 / 黑客复用。**P0-High。**

### 金线 4 — 个人中心 MVP 未提交 + 后端测试失败

**链路**：
1. 用户后端：`preferenceController.fontSize` 16.5 通过 parseInt 截断为 16 通过校验（`backend/src/controllers/preferenceController.js:56-62` vs 测试断言 `tests/controllers/preferenceController.test.js:95`）—— **明确是 bug**。
2. 用户后端：`/novels/:id/status` 路由用 `optionalAuth`（设计为允许匿名预览），但测试期望 401（`tests/integration/novelStatus.test.js:102-106`）—— **测试与产品口径冲突**。
3. 数据库：未提交两张表 `user_preferences` / `notifications` 主键 BIGINT UNSIGNED，与既有 INT UNSIGNED 不一致。
4. 用户前端：`views/profile/*` 5 个页 + 8 个组件 + 2 store + 2 api 完整实现，等待入仓。

**结论**：必须先做产品决策（status 鉴权口径）+ 修 fontSize bug + 改 BIGINT → INT，再把整批改动作为一组干净 commit 入库。**P0-High。**

### 金线 5 — Nginx 上线就裸跑 + 管理后台无入口

**链路**：
1. `nginx.conf` 仅 `listen 80`，无 443 / SSL / HSTS / CSP。
2. `nginx.conf` 没有 admin.momofx.cn 或 `/admin/` server block——管理前端 dist 上传到服务器后**默认无访问入口**，AGENTS.md:209 已警示。
3. `nginx.conf:47` 全局 `Access-Control-Allow-Origin *` + 后端 cors `credentials: true` 双层重叠。

**结论**：任何"按 deploy-guide.md 一键部署"的人会发现"管理后台进不去"。**P0-High。**

## 10. 元说明 — 为什么本次审查改成主代理顺序产出

按用户要求"安排多个角色分批去审查 / 每个子代理输出一份报告"，主代理依次：

1. 先调用 superpowers `dispatching-parallel-agents` skill，分派 6 个独立子代理（用户前端审查官 / 用户后端审查官 / 管理前端审查官 / 管理后端审查官 / DBA 迁移审查官 / 安全工程化审查官），每位都给了详细的 prompt + 期望输出路径。
2. 6 个子代理全部因推理网关 `sub2api.bbroot.com` 返回 `503 No available accounts` 失败。
3. 重派"先 1 个数据库审查官"做探测，仍 503。
4. 重派"3 个数据库 / 用户后端 / 用户前端"，仍 503。

### 6.2 未跑测试，按代码静态确认 2 个失败用例

#### 失败用例 1：preferenceController fontSize 16.5

证据链：
- 测试期望（`backend/tests/controllers/preferenceController.test.js:95`）：`expect(validateAndNormalize({ fontSize: 16.5 }).ok).toBe(false);`
- 控制器代码（`backend/src/controllers/preferenceController.js:56-62`）：
  ```js
  const n = Number.parseInt(body.fontSize, 10);
  if (!Number.isInteger(n) || n < 12 || n > 24) { return { ok:false }; }
  ```
- 行为：`Number.parseInt(16.5, 10)` → `16`，`Number.isInteger(16)` → `true`，落进 12-24 区间，**返回 ok:true**。所以测试断言失败。
- **修复方案（最小改动）**：用 `Number(body.fontSize)` 而非 parseInt：
  ```js
  const n = Number(body.fontSize);
  if (!Number.isInteger(n) || n < 12 || n > 24) { ... }
  ```
  这样 16.5 进入会得到 16.5，`Number.isInteger(16.5) === false`，被拒绝。

#### 失败用例 2：novelStatus 401

证据链：
- 测试期望（`backend/tests/integration/novelStatus.test.js:102-106`）：未带 token GET `/api/novels/1/status` → status 401。
- 路由（`backend/src/routes/novels.js:396-401`）：用的是 `optionalAuth`，匿名也能进。
- 控制器（`backend/src/controllers/novelController.js:436-446`）显式实现"匿名时返回全 false 占位 + likes/collections 计数"。
- 这是 **测试与产品口径冲突**——控制器代码是有意为之（函数注释明说"匿名访问时返回全 false / 0 的占位结果"）。
- **判断**：控制器逻辑符合"小说状态可匿名预览"的产品需求，**测试预期需要改**——将断言从 `expect(res.status).toBe(401)` 改为 `expect([200,401]).toContain(res.status)` 或直接删除该用例并补充匿名 happy 测试。负责人需要先做产品口径决策。

## 8. 安全审计

### 8.1 SQL 注入

#### 良好案例（参数化）

绝大多数都是 `pool.query(SQL, params)`：
- `authController.js:50,58,143,228,271,347` — 注册/登录/refresh/getCurrentUser/changePassword
- `notificationController.js:53-67,87-90,113-128,143-145` — 通知中心
- `userController.js:62-65,84-96,102-105` — 书架

#### 半危险（模板字符串拼接，但当前不可注入）

- `notificationController.js:54`：`SELECT COUNT(*) AS total FROM notifications WHERE ${whereSql}`，`whereSql` 来自固定数组拼接 `'user_id = ?'` / `'is_read = 0'`，**安全**。
- `userController.js:62-65`：`SELECT ... WHERE ${whereSql}`，同上。
- `novelController.js:574,591-592,620,640-641`：`${activeWhere}` / `${ratingWhere}` 来自 `hasDeletedAt ? 'AND deleted_at IS NULL' : ''`（line 73,77），**当前安全**。
- `userController.js:85-91`：`${extraSelect}` 来自 hasColumn 的字符串数组拼接，**当前安全**。

风险评估：当前不可注入，但**模板拼接 SQL** 风格本身是漏洞窗口——任何后续维护者把外部输入塞进这些变量就破。**建议后续重构成完全参数化**。

#### 真正动态字段名

- `userController.js:73`：`ORDER BY ${sortField} ${sortDirection}` —— `sortField` 来自 `SORTABLE_FIELDS[requestedSortBy]` 白名单查找，`sortDirection` 是 `'ASC'/'DESC'` 二选一。**安全**。

#### 总结

SQL 注入面：**当前无可利用注入点**，但模板拼接风格滑坡风险存在，列入 P2。

### 8.2 鉴权风险

- **缺失**：`authorController.js` 公开接口 `/authors/:authorId` 用 optionalAuth，没问题；写操作（关注/取关）走 authenticate ✅
- **优秀**：`notificationController.js:113-115` 在 markRead 前 SELECT 校验 user_id，防 IDOR ✅
- **优秀**：`bookmarks/highlights/interestTags/notifications/user` 都用 `router.use(authenticate)` 整段保护
- **风险**：`writeRateLimiter` 用 `req.user?.id || req.ip || 'anonymous'` 作为桶键（line 24），**当 trust proxy 失效**或匿名时所有匿名请求**共享同一桶**——配合 admin-backend 的 trust proxy 缺失（见报告 06），可能在生产被绕过。但用户后端 `app.set('trust proxy', 1)` 已开（`app.js:38`），**用户后端这块没问题**。

### 8.3 JWT

- 双密钥设计 ✅（access secret + refresh secret，`backend/src/config/index.js:91-95`）
- access 7d / refresh 30d ✅
- 未显式 `algorithm: 'HS256'`（默认 HS256，但显式更稳）
- `refresh` 使用 `config.jwt.refreshSecret`（authController.js:225）✅，与 admin-backend 单密钥不同

### 8.4 文件上传

- `uploads/` 显式 `Cross-Origin-Resource-Policy: cross-origin`（app.js:72-77）
- multer 配置在 uploadController.js（385 行，未细读，但根据路由 `routes/upload.js` 145 行限制了 5 个端点，全部 authenticate）
- 上传目录大小 102MB（来自报告 06）—— 可能没用 `.gitignore` 屏蔽产物目录

### 8.5 XSS / 输入清洗

- 项目依赖 `xss` ^1.0.15 + 自实现 `utils/sanitizer.js`，测试 `tests/utils/sanitizer.test.js` 覆盖在

# 数据库脚本索引

最后更新日期：2026-05-17

本文是 `docs/06-数据库脚本/` 的当前入口文档，用于说明数据库脚本的用途、执行边界和风险等级。本文只整理脚本入口，不替代 `docs/01-核心文档/` 中的数据库设计文档，也不替代 `docs/02-快速启动指南/` 中面向新人的操作步骤。

## 当前结论

- 当前目录没有接入应用启动流程的自动迁移脚本。
- 当前目录没有成体系的版本化迁移目录，例如 `migrations/`、`YYYYMMDD_xxx.sql` 或 ORM migration 文件。
- 生产、准生产或共享测试库执行任何脚本前，都应先备份数据库。
- `database_init.sql` 会 `DROP TABLE IF EXISTS` 后重建核心表，属于破坏式初始化脚本，不应对已有业务数据的数据库直接执行。
- `seed_data_complete.sql` 是当前推荐的开发种子数据入口；`test_data.sql` 和 `historical_test_data_backup.sql` 更适合作为历史测试数据参考。
- 管理员账号相关 SQL 只用于手工修复登录账号，不属于后台管理系统完整初始化流程。

## 脚本分层

| 分层 | 文件 | 执行方式 | 当前状态 | 适用场景 | 风险说明 |
|---|---|---|---|---|---|
| 初始化脚本 | `database_init.sql` | 手工执行 | 当前可用，但有破坏性 | 本地空库首次建表、重建开发库 | 会删除并重建核心表；不要用于保留数据的环境 |
| 初始化/后台扩展脚本 | `admin_tables.sql` | 手工执行 | 当前可用，需人工确认默认账号 | 给已有 `ai_xsread` 库补建后台管理表、视图、过程、触发器和索引 | 包含占位密码和 `ALTER TABLE`；重复执行前需检查索引是否已存在 |
| 自动迁移脚本 | 暂无 | 不适用 | 未建立 | 暂无 | 不要误把本目录 SQL 当成应用自动迁移 |
| 手工修复脚本 | `更新admin密码-admin123.sql` | 手工执行 | 当前可用 | 用户端 `users` 表中的 `admin` 登录密码重置为 `admin123` | 会覆盖 `admin` 用户密码；只在确认目标账号后执行 |
| 管理员脚本 | `创建管理员账号.sql` | 手工执行 | 可用但偏历史 | 创建或修复用户端 `users` 表中的 `admin` 和测试账号 | 会创建测试账号并覆盖相关账号密码；优先使用更明确的密码重置脚本 |
| 补丁脚本 | `admin_tables.sql` 内部后段 | 手工执行 | 混在后台建表脚本中 | 后台表相关视图、过程、触发器、索引补建 | 当前未拆成独立 patch 文件；执行前阅读脚本后段 |
| 开发种子数据 | `seed_data_complete.sql` | 手工执行 | 推荐入口 | 本地开发库导入完整演示数据 | 有 `ON DUPLICATE KEY UPDATE`，仍建议空库或可重置环境使用 |
| 历史测试数据 | `test_data.sql` | 手工执行/参考 | 备份参考优先 | 老版本开发测试数据、比对旧字段和旧样例 | 密码哈希为 placeholder，可能不适合登录验证 |
| 历史备份脚本 | `historical_test_data_backup.sql` | 参考，不推荐直接执行 | 由旧 `mm.md` 重命名而来 | 保留旧测试数据正文，供追溯和比对 | 原文件扩展名误为 `.md`，内容实际是 SQL；不作为当前入口 |

## 推荐执行顺序

### 本地空库首次初始化

1. 确认目标库可以清空或重建。
2. 执行 `database_init.sql` 建库建表并插入基础分类、标签、配置。
3. 需要完整演示数据时，执行 `seed_data_complete.sql`。
4. 需要后台管理系统表时，执行 `admin_tables.sql`。
5. 如管理员登录失败，再按需执行 `更新admin密码-admin123.sql` 或通过后端脚本创建管理员。

示例命令：

```bash
mysql -u toefl_user -p < docs/06-数据库脚本/database_init.sql
mysql -u toefl_user -p ai_xsread < docs/06-数据库脚本/seed_data_complete.sql
mysql -u toefl_user -p ai_xsread < docs/06-数据库脚本/admin_tables.sql
```

### 已有开发库补建后台表

1. 先确认核心表已存在，例如 `users`、`novels`、`chapters`。
2. 备份当前库。
3. 执行 `admin_tables.sql`。
4. 检查 `admin_users`、`admin_logs`、`banners`、`announcements`、`statistics_daily` 是否存在。
5. 检查 `view_admin_statistics`、`sp_log_admin_action`、`trg_banner_click` 是否创建成功。

### 只修复管理员登录

1. 确认要修复的是用户端 `users` 表里的 `admin` 账号，而不是后台专用 `admin_users` 表。
2. 执行 `更新admin密码-admin123.sql`。
3. 用用户名 `admin`、密码 `admin123` 验证。

如需创建一组本地测试账号，可改用 `创建管理员账号.sql`，但执行前要注意它会同时写入 `testuser`、`reader1`、`reader2`。

## 不建议直接执行的文件

`test_data.sql` 和 `historical_test_data_backup.sql` 仍保留，因为它们包含旧版测试数据正文，对字段演变、样例内容和历史排查有价值。但它们不再作为新环境的默认数据入口：

- `test_data.sql` 使用 placeholder 密码，登录验证价值有限。
- `historical_test_data_backup.sql` 来自旧 `mm.md`，文件内容是 SQL 测试数据，不是说明文档。
- 新环境优先使用 `seed_data_complete.sql`。

## 自动迁移边界

截至 2026-05-17，本目录没有自动迁移脚本。判断标准如下：

- 自动迁移：由应用、部署流程、ORM 或 migration runner 自动发现并按版本执行的脚本。目前本目录无此类文件。
- 手工执行：需要开发者在 MySQL 客户端、phpMyAdmin 或命令行明确执行的脚本。当前目录所有可执行 SQL 都属于这一类。
- 备份参考：保留历史 SQL 正文，但不作为当前默认执行入口的文件，例如 `test_data.sql`、`historical_test_data_backup.sql`。

未来如果新增自动迁移，请单独建立 `migrations/` 目录，并在每个文件名中包含日期或递增版本号，同时更新本文的自动迁移章节。

## 执行前检查清单

- 确认连接的是目标数据库，不是生产库或他人共享库。
- 对已有数据的数据库先执行备份。
- 阅读脚本顶部说明和末尾使用说明。
- 确认脚本中是否包含 `DROP TABLE`、`ALTER TABLE`、`UPDATE users`、默认密码或测试账号。
- 执行后检查关键表、索引、账号和演示数据是否符合预期。

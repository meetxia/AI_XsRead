## 🤝 协作规范

### Git协作规范

#### 分支策略
```
main (生产分支)
  └─ develop (开发分支)
       ├─ feature/A-* (A的功能分支)
       ├─ feature/B-* (B的功能分支)
       └─ feature/C-* (C的功能分支)
```

#### 提交规范
```bash
# 功能开发
git commit -m "feat(module): 添加XX功能"

# Bug修复
git commit -m "fix(module): 修复XX问题"

# 性能优化
git commit -m "perf(module): 优化XX性能"

# 文档更新
git commit -m "docs: 更新XX文档"

# 代码重构
git commit -m "refactor(module): 重构XX模块"

# 测试相关
git commit -m "test(module): 添加XX测试"
```

#### 合并流程
```bash
# 1. 自己的分支开发完成
git checkout feature/A-reading-system
git add .
git commit -m "feat(reading): 完成阅读系统"

# 2. 合并到develop
git checkout develop
git pull origin develop
git merge feature/A-reading-system

# 3. 推送到远程
git push origin develop

# 4. 通知团队
# 在群里通知: "阅读系统已合并到develop,请及时拉取"
```

### 代码Review规范

#### Review清单
```
□ 代码规范检查
□ 功能完整性检查
□ 性能问题检查
□ 安全漏洞检查
□ 注释完整性检查
□ 测试覆盖率检查
```

#### Review流程
1. 开发者提交PR
2. 至少1人Review
3. Review通过后合并
4. 合并者负责通知团队
